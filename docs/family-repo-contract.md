# Family Repo Contract

Each skill family lives in its own repository. `skill-tooling` operates on those external family repos.

## Required Layout

```text
family-repo/
  family.json
  source/
    skill-one.md
    skill-two.md
  overrides/
    skill-one/
      claude.md
  grok/
  grok-build/
  claude/
  claude-code/
  openai-chatgpt/
  codex/
```

## Ownership Rules

- `source/` is authored by humans.
- `overrides/` is authored by humans only when a tool needs different wording.
- Top-level tool directories are generated by `skill-tooling`.
- Local `.env` and `.skill-tooling.env` files are for secrets only and should remain untracked.

If a generated tool directory is changed by hand, the next deploy will overwrite it.

## `family.json`

`family.json` must be valid JSON with this minimum shape:

```json
{
  "skill_family": {
    "name": "customer-support",
    "description": "Reusable support skills for support teams",
    "version": "0.1.0"
  },
  "targets": [
    "grok",
    "grok-build",
    "claude",
    "claude-code",
    "openai-chatgpt",
    "codex"
  ]
}
```

`family.json` is the required canonical filename for every family repo.
Repo-specific manifest filenames like `FH-Coaches.yml` are not part of the current contract.

Rules:

- `skill_family.name` must be a lowercase slug using letters, numbers, and hyphens.
- `skill_family.description` must be non-empty.
- `skill_family.version` must be non-empty.
- `targets` must contain one or more supported target identifiers.

The machine-readable schema for this file lives at [../schemas/family.schema.json](../schemas/family.schema.json).

## Source Skill Files

Each skill lives in `source/<skill-id>.md`.

Minimum frontmatter:

```yaml
---
name: Researcher
description: Investigate and synthesize relevant information
targets: grok, claude, codex
---
```

Rules:

- `<skill-id>` comes from the filename stem.
- Skill ids must use letters, numbers, dots, underscores, or hyphens.
- `name` is optional and acts as the human-facing display name.
- `description` is required.
- `targets` is optional. If omitted, family-level targets are used.

## Overrides

When a specific tool needs different text, add:

```text
overrides/
  researcher/
    claude.md
    codex.md
```

Rules:

- `overrides/<skill-id>/<target>.md` replaces the generated text for that skill on that target.
- Override targets must also be enabled for that skill.
- Overrides are whole-file replacements for the target-specific skill text.

## Behavior

- `skill-deploy --source .` rewrites the top-level target folders in the family repo.
- `skill-deploy --repo owner/repo --publish ...` can clone a repo, generate target folders, and publish them in one run.
- Generated target folders are the manual deployment artifacts as well as the publish inputs.
